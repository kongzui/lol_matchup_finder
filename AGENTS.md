# 저장소 가이드라인

## 지침 우선순위

이 문서의 규칙은 다음 우선순위를 따른다.

1. 사용자가 현재 대화에서 직접 지시한 내용
2. 파일 작업 핵심 안전 규칙
3. 보안 및 설정 관련 규칙
4. 프로젝트 구조, 코딩 스타일, 구현 원칙
5. LLM 작업 행동 원칙

규칙이 충돌할 경우 더 구체적이고 안전한 지침을 우선한다.

## LLM 작업 행동 원칙

일반적인 LLM 코딩 실수를 줄이기 위한 행동 지침입니다.
이 지침은 프로젝트별 규칙과 함께 적용하되, 프로젝트별 규칙이 더 구체적인 경우 프로젝트별 규칙을 우선합니다.

**트레이드오프:** 이 지침은 속도보다 신중함을 우선합니다. 단순한 작업에서는 상황에 맞게 판단하세요.

### 1. 코딩하기 전에 생각하기

**추측하지 마세요. 혼란을 숨기지 마세요. 트레이드오프를 드러내세요.**

구현하기 전에:

- 가정을 명확히 말하세요. 확실하지 않다면 질문하세요.
- 여러 해석이 가능하다면 조용히 하나를 고르지 말고, 가능한 해석들을 제시하세요.
- 더 단순한 접근이 있다면 말하세요. 필요하다면 반대 의견도 제시하세요.
- 불명확한 부분이 있다면 멈추세요. 무엇이 헷갈리는지 명확히 말하고 질문하세요.

### 2. 단순함을 우선하기

**문제를 해결하는 최소한의 코드만 작성하세요. 추측성 구현은 하지 마세요.**

- 요청받지 않은 기능을 추가하지 마세요.
- 한 번만 쓰이는 코드에 추상화를 만들지 마세요.
- 요청받지 않은 “유연성”이나 “설정 가능성”을 추가하지 마세요.
- 실제로 발생할 수 없는 상황을 위한 에러 처리를 추가하지 마세요.
- 200줄로 작성했는데 50줄로 해결할 수 있다면 다시 줄이세요.

스스로에게 물어보세요: “시니어 엔지니어가 이 코드를 보고 과하게 복잡하다고 할까?”
그렇다면 단순화하세요.

### 3. 필요한 부분만 정확히 수정하기

**요청한 문제를 해결하는 데 필요한 코드만 수정하세요. 내 변경 때문에 생긴 불필요한 코드만 정리하세요.**

기존 코드를 수정할 때:

- 요청 범위 밖의 코드, 주석, 포맷을 건드리지 마세요.
- 관련 없는 코드를 “겸사겸사” 개선하지 마세요.
- 망가지지 않은 코드를 리팩터링하지 마세요.
- 기존 코드 스타일이 마음에 들지 않아도 그대로 따르세요.
- 관련 없는 죽은 코드를 발견했다면 삭제하지 말고, 작업 완료 후 따로 언급하세요.

내 변경으로 인해 불필요해진 코드가 생겼다면:

- 내가 추가하거나 수정한 결과로 사용되지 않게 된 import, 변수, 함수는 제거하세요.
- 작업 전부터 이미 존재하던 죽은 코드는 요청받지 않았다면 제거하지 마세요.

판단 기준: 변경된 모든 줄은 사용자의 요청과 직접 연결되어 있어야 합니다.

### 4. 목표 중심으로 실행하기

**작업을 시작하기 전에 성공 기준을 명확히 정의하세요.**

요청을 그대로 막연하게 처리하지 말고, 완료 조건을 구체적으로 정리하세요:

- 무엇을 바꿔야 하는지
- 무엇은 바꾸지 말아야 하는지
- 완료 후 사용자가 어떤 결과를 확인할 수 있어야 하는지

예시:

- “버그 수정” → “어떤 상황에서 발생하던 문제가 사라져야 하는지 명확히 하기”
- “UI 개선” → “어떤 화면 요소가 어떻게 바뀌어야 하는지 명확히 하기”
- “리팩터링” → “동작은 유지하면서 어떤 구조만 개선할지 명확히 하기”
- “기능 추가” → “사용자의 입력과 기대 결과를 명확히 하기”

여러 단계가 필요한 작업이라면 짧은 계획을 제시하세요:

1. [단계] → 목표: [이 단계에서 달성할 것]
2. [단계] → 목표: [이 단계에서 달성할 것]
3. [단계] → 목표: [이 단계에서 달성할 것]

성공 기준이 명확하면 불필요한 구현을 줄이고, 작업 범위를 벗어나는 변경을 피할 수 있습니다.
약한 기준, 예를 들어 “작동하게 만들기” 같은 목표는 구현 범위가 흐려지고 불필요한 변경으로 이어질 수 있습니다.

**이 지침이 잘 작동하고 있다는 신호:**
불필요한 변경이 줄어들고, 과도한 복잡성 때문에 다시 작성하는 일이 줄어들며, 실수한 뒤가 아니라 구현 전에 확인 질문이 나옵니다.

## 코딩 스타일 및 네이밍 규칙

- Python: 들여쓰기 4칸; 줄 길이 최대 100자
- 주석/독스트링/로그는 한국어로 작성, 파일은 **UTF-8 (BOM 없음)**
- 파일을 읽거나 쓸 때는 가능한 도구/API에서 UTF-8 인코딩을 명시하고, 저장 시 **UTF-8 (BOM 없음)** 을 유지할 것.
- 한글이 포함된 파일에서 셸 출력이 깨져 보여도 파일 손상으로 단정하지 말 것.
- 파일 전체 재작성은 사용자가 명시적으로 요청했거나, 파일 대부분이 실제로 바뀌는 경우에만 허용할 것.

## 구현 원칙

- 수정 범위는 요청한 문제 해결에 필요한 최소한으로 제한할 것.
- 요청한 문제와 직접 관련 없는 리팩터링, 스타일 정리, 구조 변경은 하지 말 것.
- 코드 구현은 주변 코드의 기존 양식과 패턴에 맞출 것.
- 동일한 역할의 기존 상수, 배열, 헬퍼 함수가 있으면 재사용하고, 같은 의미의 값을 인라인으로 하드코딩하지 말 것.

## 코드 수정 후 검증

- Python 파일을 수정한 경우: `ruff format .` 실행 후 `ruff check .`
- TS/TSX/CSS 파일을 수정한 경우: `cd frontend && npm run format:web` 실행 후 `npm run check:web`
- 여러 언어가 함께 바뀐 경우 수정한 언어에 해당하는 명령만 실행한다.
- Windows의 `npm.cmd`를 사용한다.

## 테스트 가이드라인

- 프로젝트 계획/문서에 명시되거나 사용자가 직접 지시한 경우에만 테스트 작성 및 실행.
- `scripts/` 내에 독립 실행 가능한 스크립트로 작성.
- **주문 관련 기능은 직접 테스트하지 말 것.** 실제 주문 동작 검증은 사용자가 직접 진행한다.

## 작업 완료 보고 가이드라인

- 코드 수정 완료 후에는 비전공자도 이해할 수 있도록 무엇이 어떻게 바뀌었는지 번호를 매겨 쉽게 설명할 것.

## Git 작업 가이드라인

- 기본적으로 커밋은 자동 수행하지 말 것.
- 코드 수정 완료 후에는 변경 내용을 요약한 추천 커밋 메시지를 한 줄로 출력할 것.
- 추천 커밋: 짧고 현재 시제 명령형 (예: "대시보드 WebSocket 재연결 로직 추가").

## 파일 작업 핵심 안전 규칙

**프로젝트 작업 디렉토리 외부의 파일/폴더는 절대 수정, 삭제, 접근하지 마십시오.**

### 절대 금지 행동:

1. **시스템 폴더 절대 접근 금지**: Git/VSCode 설치 디렉토리, `C:\Windows`, `C:\Program Files`, 사용자 AppData 폴더
2. **파일/폴더 삭제 시 명시적 확인 없이 진행 금지**: 대상 경로가 프로젝트 루트 내인지 항상 확인
3. **다음에 영향을 줄 수 있는 작업 금지**: 버전 관리 파일, IDE 설정, 프로젝트 외부 Python 패키지, 시스템 바이너리
